### Flujo de datos — POC 3 (Privy + Defindex)
```
privy-defindex-poc.ts
  → getOrCreateStellarWallet()       [privy-stellar-wallet.ts]
  → depositToDefindexVault()         [privy-defindex-wallet.ts]
      → POST api.defindex.io/vault/{addr}/deposit?network=testnet  → { xdr }
      → TransactionBuilder.fromXDR() → transaction.hash()
      → privy.rawSign(walletId, hash)  → signature
      → attach DecoratedSignature
      → POST api.defindex.io/send?network=testnet  → { txHash }
```

## Log de Decisiones

1. **Privy rawSign para Soroban** — Privy Stellar es Tier 2 (solo firma), no puede invocar contratos directamente. Se delega la construcción del XDR al API de Defindex y Privy solo firma el hash. ✅ Verificado funcional.

2. **Defindex REST API en vez de SDK** — El SDK de Defindex introduce una dependencia adicional. La API REST es suficiente: `POST /vault/{addr}/deposit` devuelve XDR listo para firmar. Menos fricción, mismo resultado.

3. **amounts como Number en vez de string** — La API de Defindex rechaza `amounts` como array de strings; espera números enteros. Se usa `Number(amountStroops)` al serializar el body.

4. **Reutilizar `getOrCreateStellarWallet()`** — Los 3 POCs de Privy-Stellar comparten la misma wallet (mismo `idempotency_key`). No se crean wallets duplicadas.

5. **Auto-fondeo vía Friendbot** — Los POCs de Stellar incluyen lógica de auto-fondeo para no requerir pasos manuales en testnet.

6. **Crossmint `contract-call` vs Privy `rawSign`** — Crossmint abstrae la invocación Soroban (maneja simulación, fee bump, auth entries). Privy solo firma el hash de la TX ya construida. El API de Defindex cubre el gap de construcción para el caso Privy.

7. **`@stellar/stellar-base` sin stellar-sdk completo** — Suficiente para parsear XDR y adjuntar firmas. Evita agregar `@stellar/stellar-sdk` (más pesado, incluye SorobanRpc que no se necesita cuando el XDR viene de la API).
